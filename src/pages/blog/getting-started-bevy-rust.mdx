## Why Rust and Bevy?

After evaluating a few options — bracket-lib, Fyrox, and Macroquad — I landed on Bevy. The ECS architecture felt like the right fit for a roguelike where you have lots of entities with overlapping behaviours. Rust's ownership model also forces you into patterns that work well with ECS.

## First Impressions of the Borrow Checker

Coming from C++ and GDScript, the borrow checker is the biggest adjustment. At first it feels like the compiler is fighting you. But after a few days it starts to feel more like a collaborator — it's catching real bugs before they happen.

The key insight is that the borrow checker pushes you toward the same patterns Bevy wants anyway. You can't easily share mutable state between systems, which means you naturally end up using components, resources, and events to communicate instead.

## Setting Up the Project

```bash
cargo new bevy_rogue
cd bevy_rogue
cargo add bevy
```

Bevy's compile times on first build are notorious. I'd recommend enabling the dynamic linking feature for debug builds — it makes iteration much faster:

```toml
[features]
default = []

[profile.dev]
opt-level = 1

[profile.dev.package."*"]
opt-level = 3
```

## ECS in Practice

The first thing that clicked for me was thinking in systems rather than objects. Instead of a `Player` class with methods, you have:

- A `Player` component (marker)
- A `Position` component
- A `Health` component
- A `move_player` system that queries for entities with both `Player` and `Position`

It feels unusual at first but the resulting code is surprisingly clean and easy to extend.
